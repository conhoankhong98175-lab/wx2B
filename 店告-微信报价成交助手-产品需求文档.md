## 1. 文档目的

本文档将「店告」从“门店海报/内容助手”转型为“面向非标服务商的微信报价成交助手”，明确产品目标、用户、范围、业务规则、功能需求、验收标准、数据指标和上线计划，供产品、设计、研发、测试、运营与商业化评审使用。

本文所称“首版”均指通过付费验证后开发的 MVP；在付费验证门槛未达到前，只允许通过表格、人工服务或无代码原型验证，不进入完整产品开发。

## 2. 项目记忆与已确认决策

### 2.1 一句话战略

> 海报只是输出，报价才是工作，成交才是价值。

### 2.2 产品定位

面向通过微信承接非标服务需求的小微商家，提供 ERP 之前的轻量移动报价前台：客户发来尺寸、数量、材料和工艺后，商家在 60—90 秒内算清价格，生成带品牌、明细、有效期和版本号的客户专属报价，并追踪客户查看、接受或申请修改的结果。

### 2.3 已确认的产品决策

| 决策项        | 结论                                                                                                                 |
| ------------- | -------------------------------------------------------------------------------------------------------------------- |
| 首发用户      | 1—20 人的广告制作、招牌、喷绘写真、图文快印门店                                                                      |
| 首要使用者    | 老板、接单员、销售人员                                                                                               |
| 核心场景      | 微信收到询价后移动端快速算价、发价、改版并确认客户接受的版本                                                         |
| 核心价值      | 少算错、少反复、快响应、可追溯、促成交                                                                               |
| 产品形态      | 商家端微信小程序 + 客户免登录报价页 + PDF/长图兜底                                                                   |
| 与 ERP 的关系 | 共存，不替代；允许导出或人工记账，不迁移库存、生产和财务数据                                                         |
| 首版公式      | 固定价、数量 × 单价、长 × 宽 × 单价                                                                                  |
| 核心闭环      | 建价格库 → 创建报价 → 分享 → 客户查看/操作 → 修改或接受 → 留痕复用                                                   |
| 北极星指标    | 每周被客户打开并产生动作的有效报价数                                                                                 |
| 收费方式      | 年订阅 + 价格库代建 + 定制公式包，不按导出次数收费                                                                   |
| 明确不做      | 库存、采购、生产排程、财务总账、通用 CRM、营销自动化、自由海报编辑器、预约排班、会员、多门店经营、首版定金与电子合同 |

### 2.4 核心假设

以下假设必须先通过真实付费和真实报价验证：

1. 目标门店每周至少有 20 次询价，并存在明显的移动报价效率问题。
2. 基础公式、附加项和手工调整可覆盖至少 70% 的高频订单。
3. 商家愿意为节省报价时间、降低错价和保存版本支付 299 元/年或 499 元起的代建费用。
4. 客户愿意打开报价页；即使不打开，PDF/长图仍能完成交付。
5. 已使用 ERP 的门店仍可能需要一个更轻、更快、适合微信会话的报价前台。

## 3. 背景与问题定义

### 3.1 背景

原“价目表/营业时间海报 → 门店内容助手”方案面临通用设计平台、商户 SaaS 和垂直行业系统的多重替代。模板、多尺寸、AI 文案和品牌套装属于成熟能力，难以形成持续付费理由。

广告制作和图文快印门店则普遍存在另一类更接近收入的问题：客户询价高度非标，价格由尺寸、数量、材料、工艺、安装、运输、税费、损耗、最低消费等多项条件共同决定。接单员常用计算器、Excel 和聊天记录临时计算，再发送一段文字或截图，容易出现耗时长、错价、版本混乱和争议无法追溯。

### 3.2 用户问题

1. **报价慢**：收到需求后需要在多个工具间切换，重复查价和计算。
2. **容易错价**：尺寸、单位、数量、损耗、最低收费、折扣和税费容易漏算或算错。
3. **表达不专业**：微信文字缺少品牌、条款、交付周期和有效期，客户难以确认范围。
4. **反复修改**：客户改尺寸或工艺后，旧报价与新报价混在聊天记录中。
5. **状态不透明**：商家不知道客户是否看过、接受还是仍有疑问。
6. **历史难复用**：相似需求无法快速复制，过往价格和条款难检索。

### 3.3 机会边界

产品只解决“收到询价到客户确认报价”这一段，不向后延伸到库存、生产、采购、开票、应收和会计，不向前扩展为通用营销获客平台。

## 4. 产品目标与非目标

### 4.1 产品目标

#### 验证期目标（第 1—14 天）

- 访谈 20 家目标门店，收集至少 50 份脱敏真实报价样本。
- 至少 10 家愿意提供真实样本。
- 至少 5 家支付 299 元预售或 499 元起的价格库代建费用。
- 至少 3 家在 7 天内重复创建 5 份以上报价。
- 报价耗时中位数相对原流程下降 60% 以上。
- 基础公式与可配置附加项覆盖 70% 以上的高频订单。

#### MVP 目标（上线后 90 天）

- 报价创建中位时间低于 90 秒，熟练用户的复用报价目标低于 60 秒。
- 新用户 7 天内重复创建报价的比例不低于 50%。
- 报价客户打开率不低于 70%。
- 打开后点击“接受报价”“有问题”或“申请修改”的比例不低于 15%。
- 付费用户平均每周创建 5 份以上报价。
- 因系统公式或版本造成的严重错价率低于 0.5%。
- 90 天内达到至少 30 个周活跃付费商家，否则不扩行业、不投放广告。

### 4.2 非目标

- 不成为广告快印行业 ERP。
- 不覆盖全部非标工艺和全部订单，首版覆盖 70% 高频场景即可。
- 不承诺自动识别客户聊天、图片或语音中的需求。
- 不自动给出市场价格，不提供跨商家价格比较。
- 不承担合同、订单、收款、开票或履约系统职责。
- 不以模板数量、海报导出量或注册用户数作为成功标准。
- 不在首版扩展美业、门窗、家居、维修等其他行业。

## 5. 用户与角色

### 5.1 目标商家画像

| 维度     | 定义                                                                      |
| -------- | ------------------------------------------------------------------------- |
| 企业规模 | 1—20 人，以单店和小团队为主                                               |
| 行业     | 广告制作、招牌、喷绘写真、图文快印                                        |
| 询价频率 | 每周至少 20 次询价                                                        |
| 当前工具 | 计算器、Excel、微信聊天、纸质价目或轻量开单软件                           |
| 关键特征 | 老板或接单员直接在微信接单，价格规则相对稳定但组合非标                    |
| 优先级高 | 无 ERP；或有 ERP 但移动报价仍然慢                                         |
| 优先级低 | 大型生产型印刷厂、深度使用行业 ERP 且满意、需复杂排产和成本核算的连锁企业 |

## 6. 典型场景与用户故事

### 6.1 典型场景

客户在微信中询问：“做一块 1.2 × 2.4 米的门头，用 XX 材料，含包边和安装多少钱？”接单员在小程序中选择产品和材料，输入尺寸和数量，勾选包边与安装，系统按面积、损耗和最低收费规则计算，接单员核对后生成报价 V1 并分享。客户打开后申请把尺寸改为 1.5 × 2.4 米。接单员复制 V1 生成 V2，修改尺寸后再次分享，V1 自动标记为“已被新版本替代”。客户接受 V2，系统记录时间、版本和计算快照。

### 6.2 核心用户故事

1. 作为老板，我希望一次录入高频产品和价格规则，之后所有报价员都能直接复用。
2. 作为报价员，我希望只输入必要参数就自动算出明细，同时能看到计算过程以便核对。
3. 作为报价员，我希望复制旧报价并只修改变化项，以减少重复录入。
4. 作为报价员，我希望把带品牌和有效期的报价分享到微信，同时可导出 PDF/长图。
5. 作为客户，我希望不登录就能看懂总价、范围、有效期和交付说明。
6. 作为客户，我希望明确接受当前版本，或说明我想修改什么。
7. 作为老板，我希望知道客户是否查看和接受，并能在争议时找回对应版本。
8. 作为老板，我希望成本价只对授权成员可见，绝不出现在客户页面或导出文件中。

## 7. 产品范围与版本优先级

### 7.1 优先级定义

- **P0**：形成报价闭环所必需，MVP 上线阻断项。
- **P1**：显著提升效率或支撑团队付费，可在首个正式版补齐。
- **P2**：验证核心闭环后再评估，不进入首版承诺。

### 7.2 功能范围

| 模块       | P0                                                       | P1                                | P2 / 暂缓                |
| ---------- | -------------------------------------------------------- | --------------------------------- | ------------------------ |
| 商家档案   | Logo、名称、联系人、默认条款、报价有效期                 | 多套条款、更多品牌样式            | 多品牌、多门店           |
| 价格库     | 手工录入、三类公式、最低收费、附加项、启停               | CSV 导入导出、批量编辑、分类排序  | 公开公式市场、智能定价   |
| 报价创建   | 客户信息、项目明细、附加费用、折扣、税费、手工调整、预览 | 常用组合、草稿自动保存、最近使用  | OCR/语音/聊天自动解析    |
| 版本管理   | 复制新版本、编号、版本快照、替代关系                     | 版本差异对比                      | 多人实时协同编辑         |
| 分享与输出 | 微信分享、私密客户页、PDF、长图                          | A4 打印样式、多主题               | 自由海报编辑器           |
| 客户动作   | 接受、有问题、申请修改、过期/撤回/替代提示               | 商家自定义问题字段                | 定金、电子签约           |
| 历史与通知 | 报价列表、搜索、状态、查看与动作通知                     | 筛选、客户维度聚合、数据导出      | 通用 CRM、营销自动化     |
| 团队       | 单账号与基础成员权限                                     | 最多 5 账号、共享价格库、数据汇总 | 复杂组织架构、多门店经营 |

库存、采购、生产排程、物流履约、应收、财务总账、发票、会员、预约、员工排班、业绩提成、营销自动化、朋友圈文案、AI 生图、自由排版、公开客户商城、首版在线收定金和电子合同均不在范围内。

## 8. 产品形态与信息架构

### 8.1 产品形态

1. **商家端微信小程序**：登录、配置价格库、创建报价、跟进状态和管理订阅。
2. **客户报价页**：通过高熵私密链接或小程序分享卡访问，不要求客户注册或授权手机号。
3. **导出文件**：PDF 与长图，适用于不愿打开链接、需归档或需打印的客户。
4. **运营后台**：账号与权益、行业预置数据、异常任务和售后处理；不作为商家日常工作入口。

### 8.2 商家端信息架构


## 9. 核心流程

### 9.1 首次使用流程

1. 微信授权登录并同意服务与隐私条款。
2. 创建商家：填写名称、联系人、电话/微信号，上传 Logo。
3. 选择行业“广告制作/图文快印”。
4. 选择“使用示例价格库”“手工创建”或“预约代建”；示例价格必须明确标注为演示数据，不得直接视为真实市场价。
5. 设置默认报价有效期、交付说明和条款。
6. 创建第一个真实报价。
7. 首次发布前完成计算明细确认与风险提示。

目标：完成基础资料后，使用示例或已代建价格库的用户 10 分钟内发出第一份报价。

### 9.2 创建并发布报价

1. 输入客户名称（必填）及可选联系方式/项目名称。
2. 从价格库选择产品，填写该公式要求的参数。
3. 勾选工艺和其他附加项，添加安装、运输等费用。
4. 设置折扣、税费或手工调整；系统实时更新明细。
5. 填写有效期、交付周期和备注。
6. 预览客户所见内容，检查计算明细。
7. 点击“发布报价”；系统冻结当前价格与规则快照，生成编号、V1 和私密访问凭证。
8. 选择微信分享、下载 PDF 或生成长图。

### 9.3 修改与版本流程

1. 已发布报价不可原地覆盖价格、项目、有效期和条款。
2. 商家点击“复制为新版本”，系统复制最新版本形成草稿 V2。
3. 修改后发布 V2；V1 标记为“已被 V2 替代”，旧链接仍可查看但不能接受或申请修改。
4. 新版本需要重新分享；所有客户动作绑定具体版本。
5. 若已接受的版本需要改变价格或范围，必须创建新版本；原接受记录保留。

### 9.4 客户查看与反馈流程

1. 客户打开分享卡或私密链接，无需登录即可查看。
2. 系统记录首次打开和最近打开时间，不记录与报价无关的个人信息。
3. 客户可执行：
   - **接受报价**：二次确认后记录当前版本、时间与访问凭证；页面提示“接受报价不等同于付款或签订正式合同”。
   - **有问题**：展开商家联系方式，并可提交一段问题说明；记录“客户咨询”事件。
   - **申请修改**：必须填写修改内容，提交后将该版本标记为“申请修改”。
4. 商家收到站内/微信服务通知；通知不可用时，首页待办仍必须可见。

### 9.5 撤回、过期与删除流程

- 商家可撤回处于有效状态的报价；撤回后客户只能看到状态和商家联系方式，不能查看价格明细或操作。
- 到达有效期次日 00:00（商家所在时区）自动过期；商家可复制为新版本并设置新有效期。
- 已发布报价默认不允许物理删除，只能归档或撤回，以保留审计链。
- 商家注销或发起数据删除时，按数据政策处理；所有外部链接应立即失效。

## 10. 业务规则

### 10.1 报价编号与版本

- 报价编号格式：`DG-YYYYMMDD-店铺短码-四位流水`，同一店铺内唯一。
- 同一报价单的版本从 V1 递增，禁止跳号和复用版本号。
- 报价编号标识一次客户需求，版本标识该需求下的一次正式报价。
- 发布时保存产品名称、单位、单价、公式、参数、条款和总价快照；以后修改价格库不得影响历史版本。
- 客户接受、咨询或申请修改均需记录所对应的报价编号和版本。

### 10.2 金额与精度

- 首版仅支持人民币，展示单位为“元”。
- 所有计算使用十进制定点数，禁止使用浮点数直接计算金额。
- 尺寸内部统一换算为米，面积精确到 4 位小数；中间金额保留 4 位小数，总额默认四舍五入到分。
- 商家可为店铺设置最终总价取整规则：不取整、到角、到元；首版不支持到十元或自定义脚本。
- 输入项必须同时展示数值和单位，厘米、毫米、米之间转换后需在确认页展示标准化结果。

### 10.3 三类基础公式

| 公式类型 | 计算规则                            | 必填参数              | 示例                            |
| -------- | ----------------------------------- | --------------------- | ------------------------------- |
| 固定价   | 固定金额                            | 固定金额              | 本次设计费 100 元               |
| 数量计价 | 数量 × 销售单价                     | 数量、计价单位、单价  | 名片 5 盒 × 30 元/盒            |
| 面积计价 | 长（米）× 宽（米）× 数量 × 销售单价 | 长、宽、数量、单价/㎡ | 喷绘 1.2m × 2.4m × 1 × 25 元/㎡ |

规则：

- 数量必须大于 0；长度、宽度必须大于 0。
- 每个产品需配置默认单位，并可限定允许输入的单位。
- 面积公式先计算实际面积，再应用损耗率：计费面积 = 实际面积 ×（1 + 损耗率）。损耗率默认 0，范围 0%—100%；面积公式金额使用计费面积计算。
- 每个项目可配置最低收费，项目基础金额 = `max(应用损耗后的公式金额, 最低收费)`。
- 计算页必须展示“公式金额”“损耗”“最低收费是否生效”，避免黑箱报价。

### 10.4 附加项

附加项用于覆膜、切割、包边、安装、运输等，不新增任意公式脚本。P0 支持：

1. 固定金额：如上门安装 200 元/次。
2. 按数量：如打孔 2 元/个 × 数量。
3. 按面积：如覆膜 5 元/㎡ × 计费面积。
4. 按关联项目金额比例：如特殊处理费为项目基础金额的 10%。

附加项必须显示名称、计价方式、参数和金额；附加项可关联某一项目，也可作为整单费用。商家可在单次报价中调整附加项金额，但系统必须标记为“本次手工调整”，不得反写价格库。

### 10.5 整单计算顺序

为确保所有端结果一致，整单按以下固定顺序计算：

1. 根据固定金额、数量或实际面积计算各项目原始值。
2. 面积项目应用损耗得到计费面积，计算公式金额，再应用项目最低收费。
3. 计算项目级与整单级附加项，得到报价小计。
4. 应用整单折扣：支持折扣率或固定优惠金额，二者单份报价只能选一种。
5. 应用手工调整：正数为加价，负数为减价，必须填写原因，得到税前交易金额。
6. 应用整单税务模式：不计税、单价含税或税费另计；同一报价只允许一个税率，范围 0%—100%。
7. 应用最终总价取整规则，得到客户应见总价；页面同时展示取整前金额，便于商家核对。

公式表达：

```text
小计 = Σ(项目基础金额 + 项目附加项) + 整单附加项
折后金额 = 小计 - 折扣金额
税前交易金额 = 折后金额 + 手工调整
税费另计时：税费 = 税前交易金额 × 税率；报价总额 = 取整(税前交易金额 + 税费)
单价含税时：报价总额 = 取整(税前交易金额)；其中税额 = 报价总额 × 税率 ÷ (1 + 税率)
不计税时：报价总额 = 取整(税前交易金额)
```

约束：税前交易金额和总价均不得小于 0；折扣或负向调整导致金额小于 0 时禁止发布。客户页必须明确展示税务模式、税率和税额。首版不处理部分项目计税、复杂混合税率或税务申报；系统中的税额仅用于报价展示。

### 10.6 成本价与毛利提示

- 价格库可选填成本价，仅供内部参考。
- 所有客户页面、分享标题、通知、PDF、长图和公开接口中不得包含成本价或毛利。
- 老板可配置报价员是否可见成本价与预估毛利。
- 若销售总价低于已录入成本，发布前强提示并要求二次确认；报价员无越权权限时不得继续发布。
- 未录入成本价时不得展示虚假毛利结论。

### 10.7 报价有效期

- 商家设置默认有效天数，初始建议值为 7 天；每份报价可修改。
- 有效期以日期展示，并明确“有效至 YYYY-MM-DD 23:59”。
- 过期版本不可接受或申请修改，但可查看已公开的报价内容并联系商家；商家撤回的报价除外。
- 客户在到期前已接受的报价保留“已接受”状态，不因到期自动改为过期。

客户页和接受确认弹窗需展示：报价用于说明当前版本的价格与服务范围；“接受报价”代表客户确认其查看与意向，不自动构成付款、电子签约或履约完成。商家可在自定义条款中补充自己的交易约定，但不得隐藏平台的基础提示。

## 12. 页面级需求

### 12.1 首页

- 首屏主按钮为“新建报价”。
- 展示最近 5 份报价及状态、客户、金额、版本和更新时间。
- 展示待处理事项：申请修改、客户问题、已接受、首次查看。
- 展示本周期已发布份数和权益上限。
- 不在首页放节日模板、内容推荐或营销活动流，避免偏离核心任务。

### 12.2 价格库页

- 默认按分类展示，支持搜索和最近使用。
- 列表显示产品名称、公式类型、销售价/单位、启停状态；成本价按权限显示。
- 编辑公式时提供即时试算区，输入示例参数即可看到计算过程。
- 删除已使用产品改为“停用”；未被使用的草稿产品才允许彻底删除。

### 12.3 报价编辑页

- 顶部固定显示客户、报价总额和保存状态。
- 项目卡片按公式动态展示字段，输入框旁始终显示单位。
- 每个项目可展开计算明细和附加项。
- 底部固定“保存草稿”“预览并发布”。
- 离开有未保存更改时必须提示。
- 发布时若服务端重算结果与端内结果不一致，阻止发布并刷新明细。

### 12.4 报价详情页

- 显示主状态、查看标签、客户动作和版本时间线。
- 主要操作按状态变化：分享、复制新版本、复制新报价、撤回、导出。
- 已发布版本只读；不得提供容易误解为原地编辑的入口。
- 时间线包括创建、发布、首次查看、咨询、申请修改、接受、替代、过期和撤回。

### 12.5 客户报价页

- 先展示状态、商家和总价，再展示明细与条款。
- 默认不展示内部计算过程、成本、毛利、创建人内部备注。
- 明细必须能区分项目、附加费用、优惠、调整和税费。
- 底部动作按钮在有效报价中保持可见；接受后显示确认结果和商家联系方式。
- 被替代时展示“此报价已有新版本，请联系商家获取”，但不得自动跳到客户未收到的新链接。

## 14. 数据埋点与指标口径

### 14.1 北极星指标

**每周被客户打开并产生动作的有效报价数**：按自然周去重统计正式报价版本，满足以下全部条件：

1. 报价版本由商家正式发布；
2. 至少发生一次经反预取过滤后的客户有效打开；
3. 同一周内至少发生“接受报价”“有问题”或“申请修改”之一。

同一版本一周内无论打开和操作多少次只计 1 份；内部预览、商家本人打开、爬虫和分享卡预取不计入。

### 14.2 核心漏斗

```text
创建草稿 → 添加有效项目 → 发布 → 分享 → 客户有效打开
→ 接受 / 有问题 / 申请修改 → 新版本或成交跟进
```

### 14.3 关键指标与口径

| 指标           | 定义                                                     |                  90 天目标 |
| -------------- | -------------------------------------------------------- | -------------------------: |
| 首份报价完成率 | 注册后 24 小时内发布首份报价的商家 / 新商家              |        ≥ 40%（验证后校准） |
| 报价创建时长   | 从点击新建到首次成功发布，排除后台停留超过 10 分钟的区间 |             中位数 < 90 秒 |
| 7 日重复使用率 | 首次发布后 7 天内再次发布的商家 / 首次发布商家           |                      ≥ 50% |
| 客户打开率     | 至少一次有效打开的版本 / 已分享版本                      |                      ≥ 70% |
| 客户动作率     | 有动作的版本 / 有效打开版本                              |                      ≥ 15% |
| 接受率         | 被接受版本 / 有效打开版本                                | 观察值，不单独作为产品承诺 |
| 周使用深度     | 付费周活商家每周发布版本数                               |                        ≥ 5 |
| 严重错价率     | 因系统计算/版本错误导致需撤回或赔付的版本 / 发布版本     |                     < 0.5% |
| 导出成功率     | 成功生成 PDF/长图任务 / 全部生成任务                     |                      ≥ 99% |

- `merchant_created`
- `price_item_created`
- `price_item_imported`
- `quote_draft_created`
- `quote_item_added`
- `quote_calculation_warning_shown`
- `quote_published`
- `quote_shared`
- `quote_exported`
- `quote_client_opened_valid`
- `quote_client_accepted`
- `quote_client_questioned`

所有事件需带 merchant_id、quote_id/version_id（适用时）、套餐、角色、来源和发生时间；不得把客户留言、电话、完整报价明细写入分析事件。

## 15. 非功能需求

### 15.1 性能与可用性

- 客户报价页在中国大陆常见 4G 网络下首屏 P95 小于 2.5 秒。
- 价格库 1,000 个产品以内，搜索结果 P95 小于 500ms。
- 发布报价的服务端计算与落库 P95 小于 2 秒。
- PDF/长图正常在 10 秒内生成；超时进入异步任务并提示用户稍后获取。
- MVP 月度服务可用性目标不低于 99.5%，计划维护除外。
- 关键写操作具备幂等键，弱网重试不得重复发布版本或重复记录接受。

### 15.2 安全与隐私

- 客户访问凭证至少具备 128 位随机强度，服务端仅保存安全摘要或等效安全存储。
- 所有网络传输使用加密通道；数据库中的联系方式、访问凭证和成本数据按敏感等级保护。
- 商家数据按 tenant_id 强隔离，每个服务端查询必须校验租户和角色。
- 客户页只返回白名单字段；成本、毛利、内部备注、成员信息不得先下发再由前端隐藏。
- 支持报价链接撤回、成员立即停用、账号设备异常登录处置和敏感操作审计。
- 收集客户信息遵循最小必要原则；客户查看报价不强制授权头像、昵称、手机号或位置。
- 提供商家数据导出、账号注销与数据删除入口；删除结果和保留例外在产品内明确说明。
- 备份文件与生产数据采用相同或更高等级的访问控制。

### 15.3 数据可靠性与审计

- 报价金额由服务端权威计算，客户端结果仅用于即时预览。
- 已发布版本的计算快照不可被价格库更新覆盖。
- 接受、撤回、替代和权限变更记录不可由普通用户删除。
- 每日备份核心业务数据；MVP 目标 RPO 不超过 24 小时、RTO 不超过 4 小时。
- 系统计算规则升级时必须保留规则版本；历史报价仍按原规则展示。

### 15.4 兼容性与易用性

- 支持当前主流微信版本及近两个主要版本，覆盖常用 iOS 与 Android 设备。
- 金额、单位、按钮状态不能仅用颜色区分。
- 所有错误提示说明“发生了什么、用户可以怎么做”，不得只显示错误码。
- 长图和 PDF 中中文字体需可稳定嵌入或使用具备授权的字体，避免缺字和版权风险。

## 17. 上线前验证与交付计划

### 17.1 阶段 G0：14 天付费验证

#### 第 1—3 天：真实样本

- 访谈 20 家门店，每家查看最近 3—5 份脱敏报价。
- 记录需求来源、步骤、耗时、工具、公式、错误和修改次数。
- 标记为“ERP 满意”“ERP 但移动报价痛苦”“Excel/聊天”三类。

#### 第 4—7 天：人工报价

- 用表格实现三类公式，人工生成品牌报价页、PDF 和长图。
- 要求商家发给真实客户并记录打开、接受或修改结果。
- 现场销售 299 元预售或 499 元代建，不把“愿意试用”当付费意愿。

#### 第 8—14 天：可复用原型

- 让已付款商家自行创建至少 5 次报价。
- 验证价格库、公式、复制报价、客户动作和移动端耗时。
- 对 3 家材料供应商测试渠道合作意愿，但不以渠道口头意向替代终端付费。

#### G0 通过门槛

必须同时达到：10 家提供样本、5 家真实支付、3 家七日内使用 5 次以上、耗时下降 60%、2 家愿替代原流程、70% 高频单可标准化。未通过则不进入 MVP 完整开发。

### 17.2 阶段 G1：MVP 开发（通过 G0 后，预计 6 周）

| 周期    | 交付内容                                   |
| ------- | ------------------------------------------ |
| 第 1 周 | 商家/账号、数据模型、价格库与三类公式原型  |
| 第 2 周 | 报价编辑、服务端计算、草稿与预览           |
| 第 3 周 | 发布、编号、版本、历史列表和搜索           |
| 第 4 周 | 客户页、客户动作、私密访问与通知           |
| 第 5 周 | PDF/长图、订阅配额、埋点、运营后台基础能力 |
| 第 6 周 | 安全测试、真实数据回放、灰度修复和上线准备 |

时间为立项估算，需在技术方案评审后确认；不得通过削减计算正确性、版本快照或访问控制来压缩工期。

### 17.3 阶段 G2：灰度上线

- 首批仅开放给已参与验证的 5—10 家付费商家。
- 使用其脱敏历史报价做回放，逐项核对系统金额和原人工金额。
- 每家至少完成 10 份真实报价，连续一周无严重错价后扩大到 30 家。
- 灰度期默认关闭行业扩展、公开投放和自定义脚本公式。

### 17.4 正式上线门槛

- P0 需求全部验收通过，无阻断级和高危安全缺陷。
- 三类公式、损耗、最低收费、附加项、折扣、税费与取整组合测试通过。
- 历史版本不可变、客户动作绑定版本、撤回即时失效测试通过。
- 真实报价回放严重错价率低于 0.5%。
- PDF/长图导出成功率达到 99%，客户页有效打开识别可用。
- 客服具备错价、链接泄露、误接受、导出失败和数据删除处理预案。

## 18. 测试重点与验收场景

### 18.1 计算组合测试

至少覆盖：

1. 三类公式的正常值、零值、负数、超大值和小数。
2. 毫米、厘米、米转换及面积精度。
3. 损耗率与最低收费分别生效、同时生效。
4. 四类附加项与多项目组合。
5. 固定优惠、折扣率、正负手工调整、含税/另计和取整。
6. 总价临界值、低于成本、总价小于零拦截。
7. 客户页、PDF、长图和商家端金额逐项一致。

### 18.2 版本与状态测试

- 发布 V1 后修改价格库，V1 不变。
- V2 草稿未发布时，V1 仍有效。
- V2 发布后，V1 显示被替代且不可接受。
- V1 已接受后创建 V2，V1 接受记录不变，V2 需重新接受。
- 报价在打开瞬间到期、撤回或被替代时，服务端拒绝过时操作。
- 重复点击接受/申请修改只生成一个有效动作。

### 18.3 权限与安全测试

- 报价员不可通过接口、导出、日志或前端数据包获取成本价。
- A 商家不能枚举或访问 B 商家的产品、报价、文件和客户动作。
- 私密凭证失效后，历史缓存和下载地址也不能继续访问敏感内容。
- 分享卡预取不计为有效打开。
- 员工停用后已有登录态和分享管理入口立即失效。

## 19. 风险与应对

| 风险             | 预警信号                       | 应对策略                                                         |
| ---------------- | ------------------------------ | ---------------------------------------------------------------- |
| 行业公式过于复杂 | 70% 高频单无法由基础公式覆盖   | 只覆盖高频单，保留手工调整；拒绝演化为任意脚本和定制 ERP         |
| 系统错价         | 计算与人工结果不一致、客户投诉 | 服务端权威计算、明细展示、二次确认、规则版本、真实报价回放与审计 |
| 商业价格泄露     | 商家不愿录价、链接外传         | 高熵私密链接、撤回、租户隔离、成本白名单输出、数据删除           |
| 客户不打开链接   | 打开率长期低于 50%             | 保留 PDF/长图；优化分享卡可信度；不把链接作为唯一交付方式        |
| ERP 快速复制     | ERP 增加轻量移动报价           | 保持 60—90 秒体验、客户动作与版本深度，提供标准导出并与 ERP 共存 |
| 需求滑向进销存   | 大量请求库存、排产、采购和财务 | 用非目标清单约束路线，以报价漏斗和付费验证排序需求               |
| 免费配额被滥用   | 多账号规避、成本异常           | 以商家和合理风控限制；不损害正常体验，不采集过量个人信息         |
| “接受”被误当合同 | 客户或商家产生争议             | 明确意向确认性质、版本和时间留痕，首版不做签约与定金             |
| 通知触达不稳定   | 微信通知授权不足               | 站内待办为主记录，服务通知为增强，报价详情保留完整时间线         |
| 代建变成重服务   | 每家均需长期人工维护           | 标准模板、导入和交付边界；以可复用规则比例评估是否继续行业       |

## 20. 停止、转向与扩展条件

### 20.1 终止或换行业条件

出现任一关键情况，应暂停该行业投入并复盘：

- 超过一半目标门店已使用行业 ERP，且对移动报价满意。
- 每家算法差异过大，基础公式和附加项无法覆盖 70% 高频订单。
- 有试用意向但 20 家中无人真实付款。
- 目标商家真实报价频率普遍低于每周 10 次。
- 商家只重视开单记账，不重视客户报价页、版本和反馈。

行业验证失败时，保留“价格库 + 快速算价 + 版本 + 客户动作”的产品核心，依次测试定制家居/门窗、工程维修和企业服务代理，不退回通用海报方向。

### 20.2 扩行业条件

同时满足以下条件后才考虑第二行业：

- 广告快印行业达到至少 30 个周活跃付费商家。
- 连续 4 周核心留存和使用深度稳定，严重错价率低于 0.5%。
- 价格库代建可在明确工时内交付，规则复用率可衡量。
- 新行业仍符合“高频非标报价、公式可标准化、微信即时响应、错价成本明显”。

## 21. 待验证而非待拍脑袋决定的问题

以下问题不阻塞人工验证，但必须在 G0/G1 中用数据确认：

1. 免费版按“报价编号”还是“正式版本发布次数”计配额更符合用户预期；当前暂定按正式版本发布次数。
2. 客户更愿意打开小程序页、H5 链接，还是直接查看长图/PDF。
3. 固定价是否还需要同时支持“固定单价 × 数量”；当前固定价定义为整项固定金额，有重复数量时使用数量计价。
4. 按面积附加项应使用实际面积还是含损耗计费面积；当前默认跟随主项目计费面积，并在配置时明确。
5. “接受报价”是否需要客户输入姓名或手机号；当前坚持免登录、免强制留资，以减少阻力。
6. 年度 500 份额度对高频个人商家是否过低；需依据真实发布量调整，不改为按次付费。
7. 客户问题与修改说明是否需要图片附件；首版仅文本，验证真实频率后决定。
8. 何种标准导出最能与现有 ERP 共存；首版先 PDF/CSV，不承诺双向接口。

## 22. 评审清单

### 产品评审

- 核心路径是否稳定落在“微信询价 → 90 秒报价 → 客户动作”。
- 每项 P0 是否直接服务效率、正确性、版本或成交反馈。
- 是否有任何库存、生产、财务、CRM 或自由设计需求越界。

### 设计评审

- 报价员能否在移动端单手完成主要操作。
- 数值、单位、最低收费、损耗和税费是否可理解。
- 客户能否在首屏确认商家、版本、状态和总价。

### 技术评审

- 服务端计算、定点精度、规则版本与历史快照方案是否可靠。
- 多租户隔离、成本字段保护、私密链接和撤回机制是否闭环。
- PDF/长图与客户页是否共享同一份服务端报价数据，避免金额分叉。

### 测试评审

- 是否具备公式组合、状态竞争、弱网幂等、权限越权和历史回放用例。
- 是否能判定严重错价率并追溯具体规则版本。

### 运营与商业评审

- 是否已拿到真实样本和真实付款，而非口头意向。
- 代建服务是否有标准交付边界和工时记录。
- 客服是否能处理错价、泄露、误操作和数据删除请求。

## 23. 最终验收定义

MVP 不是“能够生成一张报价图片”，而是以下闭环全部成立：

1. 商家能建立并复用自己的价格库。
2. 接单员能在移动端用明确公式和附加项快速算价，并看懂计算明细。
3. 系统能冻结每次正式报价的编号、版本、规则与金额快照。
4. 客户无需注册即可安全查看，并能接受、咨询或申请修改。
5. 商家能收到动作、创建新版本并追溯客户接受的具体版本。
6. 链接、PDF 和长图金额一致，成本等内部数据不泄露。
7. 真实用户数据证明报价更快、重复使用、有人付费，且严重错价率处于目标内。

只有当这七项同时成立，产品才完成从“做图工具”到“报价成交助手”的转型。
